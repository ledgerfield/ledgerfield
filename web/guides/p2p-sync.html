P2P ruleset sync

Share, verify and adopt tax rulesets across devices — without a central server, without exposing your books.

Tab 7 — P2P Offline-first CID-verified No PII on the wire

What travels over the network — and what never does

LedgerField's privacy boundary is strict: raw journal entries, payslips, and vault contents stay on your device forever. The knitweb layer carries only two kinds of public artefact: tax rulesets (logic files that encode rates and thresholds for a jurisdiction) and anonymised data-market aggregates. Your books are never one of them.

Guarantee: the app works entirely offline. The relay is optional convenience — you can import a ruleset from a USB stick or a local file just as easily.

Your node identity

When you open the P2P tab for the first time, LedgerField generates a node ID: a SHA-256 hash derived from a locally-generated random seed. It contains no personally identifiable information — no IP address, no device fingerprint, no account name. The ID is shown as a hex string at the top of the tab. It is stable across sessions (stored in the encrypted vault) and is the only identifier peers ever see.

Your node ID changes if you reset the vault or restore from a backup with a different seed. Old peers will treat the new ID as a new node — there is no account linkage to break.

Available rulesets

The centre of the P2P tab is the rulesets table. Each row is a ruleset that is either built-in, previously imported, or discovered from a peer. Tax rulesets in LedgerField are knitwebs — self-contained, content-addressed objects. Every column means something concrete:

Column What it tells you
Ruleset Human-readable name, e.g. NL-2025 or UK-2024
Jurisdiction ISO country code the ruleset covers
CID Content identifier — a CID that is recomputed from the ruleset bytes on every load; any tampering changes the hash
Source Built-in, peer (node ID prefix), or local file
Status Active (applied to Taxes/Filing tabs) or Available (ready to activate)

CIDs are computed using canonical encoding — keys sorted, whitespace normalised — so the same logical ruleset always produces the same identifier regardless of which tool created it.

Importing a peer ruleset

A peer shares a ruleset as a JSON file (or sends its CID over any channel you trust). Import it in three steps:

  1. In the P2P tab, click Import ruleset and select the JSON file (or paste the CID to fetch it from the relay).
  2. LedgerField recomputes the CID from the file bytes using canonical encoding. If the computed CID does not match the one you were given, the import is rejected and an error is shown — the file was altered in transit or at rest.
  3. If the CID matches, the ruleset appears in the table with source peer. Click Activate to apply it to the Taxes and Filing tabs immediately.
You can hold multiple rulesets (e.g. NL-2025 and UK-2024) and switch the active one per entity. Only one ruleset is active per entity at a time.

Publishing a ruleset

If you have customised thresholds or authored a ruleset for a new jurisdiction, you can share it with peers:

  1. In the P2P tab, click Publish ruleset and select the ruleset you want to share.
  2. LedgerField serialises it with canonical encoding and computes its CID. The CID is displayed so you can share it out-of-band (email, chat, printed QR).
  3. A JSON file is exported to your downloads folder. Share this file directly with peers, or click Announce to relay to make it discoverable on the network.
Publishing a ruleset does not reveal anything about your books. The JSON contains only tax logic (rates, brackets, credits, deadlines) — no entity name, no transaction data.

How rulesets spread: gossip over the relay

When two or more nodes are online, rulesets propagate via gossip: each node periodically tells its peers which CIDs it holds; peers request any CIDs they lack. This happens through the public relay at 5mart.ml, which is implemented on the pulse network. The relay brokers connections and caches announced CIDs — it never decrypts or inspects content.

🔁

Gossip, not broadcast

Nodes exchange CID lists, not full rulesets. A ruleset is only transferred when a peer explicitly lacks it, keeping bandwidth minimal.

🔒

Relay sees only CIDs

5mart.ml routes by content identifier. It never sees which jurisdiction you operate in, and certainly not your financials.

📴

Fully offline fallback

If the relay is unreachable, all existing rulesets remain available. Import and export via local file work without any network access.

✅

CID is the trust anchor

Because every ruleset carries its own verifiable CID, you do not need to trust the relay or the sender — only the hash function.

Quick reference

Import checklist

  1. Obtain JSON file + expected CID from peer
  2. P2P tab → Import ruleset → select file
  3. Verify: computed CID must match expected CID
  4. Activate on the relevant entity

Publish checklist

  1. P2P tab → Publish ruleset → select ruleset
  2. Copy the generated CID and share it with peers
  3. Download the exported JSON and distribute
  4. Optionally announce to relay for auto-discovery
